# Frontend Migration вЂ” vanilla в†’ Svelte 5 + Vite (planned for v0.2)

## Why migrate

The v0.1 frontend is vanilla HTML/CSS/JS served as static files. This works
and ships without a build step, but has limits:

- `innerHTML` rewrites every 5s вЂ” coarse, no diffing
- No state management for the modal, sliders, language switcher
- No type safety for the API contract
- Hard to add interactive components (charts library, complex forms)
- No tree-shaken bundle вЂ” every byte is shipped

For v0.2, we plan to migrate to **Svelte 5 + Vite**.

## Why Svelte 5 (not Vue / React)

| Criterion | Vanilla (v0.1) | Svelte 5 | Vue 3 | React |
|---|---|---|---|---|
| Runtime bundle | 0 KB | ~5 KB | ~30 KB | ~45 KB |
| Build step | none | required (Vite) | required | required |
| Reactivity model | manual | `$state` runes (signals-like) | refs + reactivity | hooks + virtual DOM |
| SFC syntax | n/a | `.svelte` (HTML + JS + CSS) | `.vue` (SFC) | JSX |
| Type safety | none | TypeScript first-class | TypeScript good | TypeScript good |
| Compile-to-vanilla | n/a | yes | partial | no |
| Learning curve | n/a | shallow | shallow | moderate |

Svelte 5's **runes** (`$state`, `$derived`, `$effect`) give signals-style reactivity
without ceremony, and the compiled output is small enough to keep the dashboard
feeling lightweight on a Pi or older eGPU host.

## Build pipeline

Vite outputs to `../src/gpu_dashboard/static/`, so the Python server doesn't
need to change вЂ” it still serves the `static/` directory.

```bash
cd frontend
pnpm install
pnpm build          # one-shot: writes to ../src/gpu_dashboard/static/
pnpm dev            # dev server on :5173 with HMR, proxies /api/* to :9999
```

`pnpm dev` should proxy `/api/*` to the Python backend so the frontend dev
loop is fast (no Python restart on UI changes).

```ts
// vite.config.ts (skeleton)
import { defineConfig } from 'vite';
import { svelte } from '@sveltejs/vite-plugin-svelte';

export default defineConfig({
  plugins: [svelte()],
  build: {
    outDir: '../src/gpu_dashboard/static',
    emptyOutDir: true,
  },
  server: {
    proxy: { '/api': 'http://127.0.0.1:9999' },
  },
});
```

## State management with Svelte 5 runes

A typed live-data store:

```ts
// src/lib/stores.ts
import type { State } from './api';

export const live = $state<State | null>(null);

let timer: ReturnType<typeof setInterval> | null = null;

export function startPolling(interval = 5000) {
  if (timer) return;
  const tick = async () => {
    try { live.value = await fetch('/api/state').then(r => r.json()); }
    catch { /* keep last good */ }
  };
  tick();
  timer = setInterval(tick, interval);
}
```

Used in a component:

```svelte
<!-- components/cards/GpuCard.svelte -->
<script lang="ts">
  import { live } from '$lib/stores';
  const g = $derived(live?.gpu);
</script>

{#if g?.alive}
  <div class="card">
    <h2>{$_t('card.gpu')}</h2>
    <div class="big" style:color={tempColor(g.temp)}>{g.temp}В°C</div>
  </div>
{/if}
```

## i18n with typed keys

```ts
// src/lib/i18n/index.ts
import en from './en.json';
import fr from './fr.json';
import { derived, get, writable } from 'svelte/store';

const dicts = { en, fr } as const;
export type Lang = keyof typeof dicts;
export type Key = keyof typeof en;

export const lang = writable<Lang>(
  (localStorage.getItem('lang') as Lang) || 'en'
);

lang.subscribe(v => localStorage.setItem('lang', v));

export const t = derived(lang, $lang => (key: Key, vars?: Record<string, unknown>) => {
  let s: string = dicts[$lang][key] ?? dicts.en[key] ?? key;
  if (vars) for (const k in vars) s = s.replaceAll(`{${k}}`, String(vars[k]));
  return s;
});

// Convenience for template usage: {$_t('card.gpu')}
export const _t = t;
```

## Migration steps

1. **Bootstrap** `frontend/` with `pnpm create vite@latest -- --template svelte-ts`.
2. **Port `i18n/en.json` and `fr.json`** from `static/i18n.js` (the keys are
   already named, just split into 2 JSON files).
3. **Wire `stores.ts`** with `live = $state(...)` polling `/api/state`.
4. **Port components one by one**, in this order:
   - `Header` + `Toast` (simplest)
   - `GpuCard`, `PowerCard`, `VramCard` (read-only)
   - `FansCard` with the SVG fan animation
   - `CoolingChart` + `PowerChart` (the most complex вЂ” reuse the Catmull-Rom
     code from v0.1's `app.js`)
   - `SettingsModal` with all 6 sub-sections
5. **Add CI step** that runs `pnpm build` and commits the diff (or uses a
   GitHub Action that builds on PRs and pushes to a `gh-pages-frontend` branch).
6. **Keep `static/i18n.js` + `static/app.js`** until v0.2 ships, then delete.

## What we keep from v0.1

- All Python backend code (`server.py`, `api.py`, `modules/*`) вЂ” unchanged.
- Profile JSON schema + validation вЂ” unchanged.
- `install.sh` вЂ” unchanged.
- All 178+ tests вЂ” unchanged.

The migration is **frontend-only**. The HTTP contract (`/api/state`,
`/api/set-power-limit`, etc.) stays identical, which makes the swap a
nearly drop-in replacement of the `static/` directory contents.

## When to merge

- After v0.1.x has been published for at least 2 weeks (let people use the
  vanilla version, gather feedback)
- When at least one of these is true:
  - Someone opens an issue asking for a more polished UI
  - We want to add a feature that's hard in vanilla (drag-and-drop fan curve,
    live preview of clock offsets before apply, drawer for OcuLink event log)
  - The project hits 100+ stars (signal that polish ROI is worth it)
